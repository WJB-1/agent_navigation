### **🏁 Phase 3 核心验收标准清单 (空间逻辑中间件)**

#### **准备工作 (测试环境配置)**

* \[ \] 确认后端 .env 文件中已经正确配置了 AMAP\_WEB\_KEY（注意：Web 服务 Key 与前端 JS API Key 通常是两个不同的 Key，需在高德控制台单独申请）。  
* \[ \] 重启后端 Node.js 服务，确保无报错日志。

#### **1\. 路由流水线连通性测试 (对应模块 3.1 & 3.3)**

**测试动作**：向后端发送一个 POST 请求，模拟前端的调用。 你可以使用以下 cURL 命令或在 Postman 中构造请求 ：

Bash

curl \-X POST http://localhost:3002/api/navigation/preview \\  
\-H "Content-Type: application/json" \\  
\-d '{"origin": "116.434307,39.90909", "destination": "116.434446,39.90816"}'

* \[ \] **HTTP 状态码**：接口应当返回 200 OK。  
* \[ \] **高德数据获取**：如果高德 Key 配置错误或路线无法规划，接口应返回清晰的错误提示（如 {"success": false, "message": "高德路径规划失败..."}），而不是直接让 Node.js 进程崩溃报错（验证容错性）。

#### **2\. 空间降噪与过滤测试 (对应模块 3.2 步骤 1\)**

**测试动作**：检查接口返回的 JSON 结构。

* \[ \] **IR 格式吻合度**：返回体必须严格遵循我们定义的 Intermediate Representation (IR) 格式，包含 route\_summary 和 key\_nodes 两个核心对象 。

* \[ \] **过滤逻辑生效**：检查 key\_nodes 数组。如果一段长距离中包含多个只是为了绘制地图折线而生成的纯直行 steps（且无特殊无障碍特征），它们应当被中间件果断抛弃或合并，key\_nodes 中仅应保留发生了“左转”、“右转”等 action 变化的节点 。

#### **3\. 时钟方位算法精准度测试 (对应模块 3.2 步骤 2 & 3\)**

**测试动作**：这是整个系统防范大模型幻觉的生命线，需要肉眼核对 JSON 里的算法结果 。 检查返回的 key\_nodes 中的第一个转向节点：

* \[ \] **方向提取**：确认 JSON 节点中存在类似 "clock\_direction": "X点钟方向" 的强语义字段 。

* \[ \] **数学逻辑推演（以高德测试数据为例）**：  
  * 假设起步阶段路段 A1 走向为“南”（转化为角度 180 度） 。

  * 到达节点后，下一路段 A2 走向为“东”（转化为角度 90 度） 。

  * 算法执行：$\\Delta \\theta \= (90 \- 180 \+ 360\) \\pmod{360} \= 270$ 度 。

  * 时钟映射：$270 / 30 \= 9$。  
  * **结果核验**：该节点的 clock\_direction 必须精准输出为 **"9点钟方向"**（即向左转）。请验证代码是否输出了这个正确的计算结果 。若输出错误，说明 Agent 写的算法逻辑有 Bug，必须打回重写。
